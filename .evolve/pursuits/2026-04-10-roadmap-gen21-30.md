## Gen 21: Parallel Exploration + Evidence Accumulation
**The multi-page research agent**

### What
- Goal decomposer: 1 cheap LLM call decides if the goal splits into sub-goals
- Parallel tab execution: independent sub-goals run in separate browser tabs
- Evidence accumulator: structured JSON tracks partial results across turns
- Smart trigger: only fires when goal has comparison/multi-item signals

### Trigger logic (NOT always-on)
```
"look up definition of X"           → single-page, no decomposition
"find 5 salons rated > 4.8"         → single-page + evidence tracking
"compare prices on Amazon vs BestBuy" → 2 parallel tabs
"plan a trip: flights + hotel"       → 2+ parallel sub-goals
```

### Impact
| Metric | Before | After |
|--------|--------|-------|
| Pass rate | 88% | 92-94% |
| Cost | $0.20 | $0.25 (more tabs, but higher pass = lower cost-per-pass) |
| Wall time | 60s | 40-45s (parallel cuts comparison tasks in half) |
| Capability | Single-page automation | **Multi-page research agent** |

### Risk: Medium
Tab management complexity, race conditions between sub-agents. Mitigated: the driver already handles popup adoption and multi-page.

## Gen 22: Open Model Cascade
**Self-hosted executor, proprietary planner**

### What
- **Planner**: stays on gpt-5.4 (needs top-tier vision + reasoning for plan generation)
- **Executor**: fine-tuned open VLM (Qwen2.5-VL-7B or Kimi K2) for deterministic step execution
- **Trajectory collection**: every successful run saves clean training data automatically
- **Fallback chain**: open model → gpt-5.4 DOM → gpt-5.4 vision (escalate on failure)

### Model candidates
| Model | Size | Why | Deploy |
|-------|------|-----|--------|
| **Qwen2.5-VL-7B** | 7B | Best open VLM at size, strong vision understanding | Single A100, or Together/Fireworks |
| **Kimi K2** | MoE | Strong reasoning, competitive with proprietary | Heavier infra |
| **Llama 4 Scout** | 17B active (109B total) | Meta ecosystem, multimodal native | MoE needs good infra |
| **Gemma 3 27B** | 27B | Google quality, well-documented | Single A100 |

### Training data pipeline
```
Successful Gen 20-21 runs
  → Extract: (goal, page_state, action, result) tuples
  → Filter: only tasks with verified correct results
  → Format: conversation-style JSONL for fine-tuning
  → Target: 10k+ trajectories across all 15 WebVoyager sites
  → Fine-tune: LoRA on Qwen2.5-VL-7B (4-8 GPU hours, ~$30)
```

### Impact
| Metric | Before | After |
|--------|--------|-------|
| Pass rate | 92% | 92% (same — executor follows plans) |
| **Cost** | **$0.20** | **$0.03-0.05** (4-7× cheaper) |
| Wall time | 45s | 30s (smaller model = faster inference) |
| **Ownership** | Vendor-locked | **Own the weights, deploy anywhere** |

### Why NOT OpenAI fine-tuning
- Don't own the weights → vendor lock-in
- Can't deploy on own infra → no cost control at scale
- $25/1M training tokens → expensive iteration
- Open models are competitive for execution (not planning)

## Gen 23: Set-of-Marks (SoM) Visual Grounding
**RECOVERED FROM GEN 50 IDEAS — should have built earlier**

### What
Overlay numbered labels on interactive elements in the screenshot before sending to the LLM. Instead of the model guessing coordinates ("I think the search box is at approximately (200, 50)"), it sees labeled elements ("Element [3] is the search box") and says "click [3]".

### Why we discarded it
Initially thought coordinate prediction from Claude's computer-use training was sufficient. But Google Flights at 50% shows coordinate accuracy is a bottleneck on complex UIs.

### How it works
1. Before screenshotting, inject a DOM overlay that draws numbered badges on interactive elements
2. Badge position = element's bounding box center
3. Screenshot captures the badges
4. The prompt says "click element [N]" instead of "clickAt(x, y)"
5. The runner maps [N] → element bounding box center → mouse.click

### Impact
| Metric | Before | After |
|--------|--------|-------|
| **Pass rate** | 92% | **95%+** (coordinate accuracy was killing Google Flights) |
| Cost | $0.04 | $0.04 (same — no extra LLM calls) |
| Wall time | 30s | 31s (+1s for badge overlay injection) |

### Why this is high-priority
Google Flights failures are mostly "stuck in date picker" — the model can SEE the right date in the screenshot but clicks the wrong coordinates. SoM eliminates coordinate guessing entirely. Magnitude doesn't use SoM — this could be our differentiation.

## Gen 24: Action Replay + Error Recovery
**RECOVERED — mistake correction through trajectory replay**

### What
When an action fails or produces unexpected results, instead of just injecting feedback text, the agent can "undo" by navigating back and replaying from a known-good checkpoint.

### How
- Runner saves page state + URL at each successful step as a checkpoint
- On failure: navigate back to the last checkpoint URL
- Re-observe from the clean state
- The feedback includes: "Your action at turn N failed. You're back at the state from turn N-1. Try a different approach."

### Why we need it
21% of Gen 15 failures were "wrong answer" — the agent went down a wrong path and couldn't recover. Currently recovery = inject feedback + hope. With checkpoints, recovery = go back to last known-good state + try differently.

### Impact
| Metric | Before | After |
|--------|--------|-------|
| Pass rate | 95% | 96-97% |
| Cost | $0.04 | $0.05 (replay costs extra turns) |
| Wall time | 31s | 35s (replay adds time) |

## Gen 25: Streaming Result Extraction
**RECOVERED — extract data AS you navigate, not at the end**

### What
Instead of navigating to a results page and THEN extracting data, extract data from every page the agent visits. Build up a structured evidence object continuously.

### Why
Many failures happen because the agent finds the data on page 3, navigates to page 4 for "confirmation," and loses the data. If extraction happens continuously, the data is never lost.

### How
- After every action, check: does the current page contain data relevant to the goal?
- If yes, extract it via runScript and append to the evidence accumulator
- The completion step reads from the accumulator, not from the current page
- Zero extra LLM calls — extraction is runScript-based

### Impact
| Metric | Before | After |
|--------|--------|-------|
| Pass rate | 97% | 97-98% |
| Cost | $0.05 | $0.05 (runScript is free) |
| Capability | **Never loses found data** |

## Gen 26: Site Pattern Learning (Live, No Fine-Tuning)
**RECOVERED — runtime memory, not trained knowledge**

### What
After completing a task on a site, save a structured pattern file:
```json
{
  "domain": "booking.com",
  "patterns": {
    "cookieBanner": {"action": "clickAt", "x": 512, "y": 680, "label": "Accept cookies"},
    "searchForm": {"type": "fill", "fields": {"destination": "@ref1", "checkin": "URL param"}},
    "datePickerBypass": "URL: booking.com/searchresults?checkin=YYYY-MM-DD&checkout=YYYY-MM-DD"
  },
  "lastUpdated": "2026-04-10"
}
```

On next visit, inject as extraContext. No fine-tuning needed — pure prompt context.

### Impact
| Metric | Before | After |
|--------|--------|-------|
| Pass rate | 97% | 98% (known patterns skip trial-and-error) |
| Wall time | 35s | **15-20s on revisits** (skip cookie banners, know form layout) |
| Cost | $0.05 | **$0.02 on revisits** (fewer turns needed) |
